# Тестирование и контроль качества

Документ описывает запуск автотестов, статического анализа и устройство CI. Перед прогоном необходимо развернуть проект по инструкции SETUP.md.

## 1. Инструменты проверки

* Стиль кода: Laravel Pint (проверка соответствия стандарту PSR-12).
* Статический анализ: PHPStan и Larastan, максимальный уровень 9 (контроль типов, контрактов и запрет mixed).
* Автотесты: PHPUnit 11 через встроенную команду php artisan test.
* Непрерывная интеграция: GitHub Actions (автоматический запуск Pint, PHPStan и тестов при push и PR).

## 2. Запуск проверок в Docker

Выполняется из корня проекта при запущенных контейнерах.

Проверка стиля кода (Pint):
docker compose exec app vendor/bin/pint --test

Запуск статического анализа (PHPStan):
docker compose exec app vendor/bin/phpstan analyse --no-progress --memory-limit=512M

Запуск автоматических тестов (PHPUnit):
docker compose exec app php artisan test

Полный прогон всех проверок одной командой (остановка при первой ошибке):
docker compose exec app sh -c "vendor/bin/pint --test && vendor/bin/phpstan analyse --no-progress --memory-limit=512M && php artisan test"

## 3. Состав и запуск автотестов

Ожидаемый результат полного прогона: 50 тестов, 0 падений, 0 пропусков.

Структура тестов:
* Feature (tests/Feature/TaskApiTest.php) — проверка эндпоинтов API: CRUD, поиск, стратегии сортировки, пагинация, ошибки 422 и 404.
* Feature (tests/Feature/TaskListCacheTest.php) — интеграция с Redis: кэширование списков задач и инвалидация кэша при мутациях.
* Unit (tests/Unit/*) — тесты сервиса, репозитория, DTO, стратегий сортировки и нормализации категорий; часть стратегий проверяется через SQLite in-memory.

Выборочный запуск:
* Запуск только тестов API:
  docker compose exec app php artisan test tests/Feature/TaskApiTest.php
* Запуск конкретного теста по его названию:
  docker compose exec app php artisan test --filter=test_can_create_task

## 4. Критерии успешного прохождения

### Laravel Pint
* Успех: Код выхода 0. В консоли выводится строка "PASS ... 75 files".
* Ошибка: Код выхода 1. Выводится список файлов с нарушениями стиля.

### PHPStan
* Успех: Код выхода 0. В консоли выводится строка "[OK] No errors".
* Ошибка: Код выхода 1. Выводится список файлов с указанием строк и характера ошибок типов.

### Автотесты (php artisan test)
* Успех: Код выхода 0. Все тесты отмечены зеленым маркером, в конце выведено "Tests: 50 passed".
* Ошибка: Код выхода 1. Тест отмечен красным маркером FAIL, выводится блок FAILED с подробным описанием несовпадения ожидаемого и фактического ответов.
* Важно: Пропуски (skipped) не допускаются. Тесты кэша требуют работающий Redis, который уже настроен внутри Docker.

## 5. Ожидаемое поведение API в тестах

* Создание задачи: HTTP-код 201 Created, тело ответа содержит id и сообщение об успехе.
* Получение и список задач: HTTP-код 200 OK, тело содержит обязательные поля схемы Task из openapi.yaml (id, title, description, due_date, create_date, status, priority, category).
* Обновление и удаление: HTTP-код 200 OK, тело содержит сообщение об успешном выполнении.
* Невалидные данные: HTTP-код 422 Unprocessable Entity, структура содержит message и массив errors.
* Несуществующая задача: HTTP-код 404 Not Found, тело ответа: {"message": "Задача не найдена."}.
* Нечисловой id в пути: HTTP-код 404 Not Found, JSON с полем message (ошибка маршрутизации, не «Задача не найдена.»).
* Пагинация списка: Заголовки ответа содержат метаданные X-Total-Count, X-Per-Page, X-Current-Page, X-Last-Page.

## 6. Конфигурация тестового окружения

Настройки зафиксированы в файле phpunit.xml:
* База данных: Используется изолированная СУБД SQLite в оперативной памяти (:memory:). Перед каждым feature-тестом вызывается трейт RefreshDatabase.
* Кэширование: По умолчанию в общих тестах кэш отключен (CACHE_TASK_LIST_ENABLED=false, CACHE_STORE=array). Внутри TaskListCacheTest драйвер принудительно переключается на Redis.
* Вне Docker: При запуске на хост-машине без расширения redis тесты кэша завершатся ошибкой. Прогон необходимо выполнять внутри контейнера.

## 7. Работа CI (GitHub Actions)

Скрипт пайплайна расположен в .github/workflows/ci.yml. Он автоматически запускается при каждом push и pull request в ветку main.

Последовательность шагов сборки:
1. Инициализация окружения (Checkout кода, установка PHP 8.3 и расширений mbstring, pdo_mysql, zip, bcmath, redis).
2. Установка зависимостей через composer install.
3. Подготовка приложения (копирование .env, генерация ключа key:generate, запуск миграций). В качестве сервисов для джобы поднимаются контейнеры MySQL и Redis.
4. Выполнение шага Pint (vendor/bin/pint --test).
5. Выполнение шага PHPStan (vendor/bin/phpstan analyse).
6. Выполнение шага Tests (php artisan test). В phpunit.xml для изоляции feature-тестов кэш по умолчанию отключён (array); TaskListCacheTest переключает Redis через config(). Переменные CACHE_STORE=redis в CI обеспечивают доступность Redis для cache-тестов.

Результаты CI доступны на GitHub во вкладке Actions. Зеленая галочка означает успешное прохождение всех проверок, красный крест указывает на падение конкретного шага. Локальный запуск трех базовых команд в Docker полностью эквивалентен логике CI.
