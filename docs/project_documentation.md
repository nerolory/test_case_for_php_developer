## **1\. ВТЗ (Высокоуровневое техническое задание)**

### **1.1. Назначение и цели**

Проект представляет собой REST API для управления списком задач. Основная цель — предоставить надежный, масштабируемый и документированный интерфейс для интеграции с клиентскими приложениями. Система позволяет создавать, читать, обновлять, удалять задачи (CRUD), а также осуществлять поиск и сортировку с поддержкой постраничного вывода.

### **1.2. Функциональные требования**

Система должна обеспечивать следующий набор бизнес-функций:

* **Создание задачи:** Фиксация новой задачи со следующими атрибутами:  
  * **Название:** строка, обязательно, до 255 символов.  
  * **Описание:** текст, опционально, без ограничений.  
  * **Срок выполнения (due\_date):** дата и время.  
  * **Дата создания (create\_date):** дата и время (доменная дата, передается клиентом).  
  * **Статус:** перечисление (не выполнена / выполнена).  
  * **Приоритет:** перечисление (низкий / средний / высокий).  
  * **Категория:** строка (например, "Работа", "Дом").  
* **Просмотр списка задач:** Получение массива задач с поддержкой:  
  * **Поиска:** по подстроке в названии задачи.  
  * **Сортировки:** по дате выполнения (due\_date) или дате создания (created\_at). По умолчанию — по id по убыванию.  
  * **Пагинации:** постраничный вывод списка. Данные передаются в теле ответа, метаданные (текущая страница, всего записей) — в HTTP-заголовках.  
* **Просмотр конкретной задачи:** Получение полной информации по идентификатору.  
* **Редактирование задачи:** Полное обновление всех изменяемых полей существующей записи.  
* **Удаление задачи:** Физическое удаление записи из системы.

### **1.3. Нефункциональные требования**

* **Документация:** Наличие интерактивной документации Swagger (OpenAPI 3.0).  
* **Развертывание:** Система должна запускаться из Docker-контейнера одной командой с минимальной конфигурацией.  
* **Качество:** Покрытие функционала автотестами, соблюдение стандартов чистого кода.

## **2\. НТЗ (Низкоуровневое техническое задание)**

### **2.1. Стек технологий**

* **Язык программирования:** PHP 8.3  
* **Фреймворк:** Laravel 12  
* **СУБД:** MySQL 8  
* **Кэш:** Redis 7 (`CACHE_STORE=redis`)  
* **Окружение:** Docker (nginx \+ php-fpm \+ mysql \+ redis)  
* **Документация:** l5-swagger  
* **Тестирование:** Pest PHP

### **2.2. Архитектура приложения**

Проект построен на основе слоистой архитектуры Laravel с жестким разделением зон ответственности (принципы SOLID, DRY):

* **Слой маршрутизации и HTTP (Controllers):** Тонкие контроллеры. Получают запрос, вызывают сервисы и возвращают API Resource. Запрещена бизнес-логика и прямая работа с БД.  
* **Слой валидации (FormRequests):** Изолированная валидация входящих данных. Обращения к БД из этого слоя запрещены (все проверки консистентности вынесены в сервис).  
* **Слой передачи данных (DTO \- Data Transfer Objects):** Данные между слоями (из контроллера в сервис) передаются исключительно через строго типизированные объекты (app/Data/), а не массивами. Реализован метод toArray() для безопасного маппинга.  
* **Слой бизнес-логики (Services):** Реализует основные сценарии. Взаимодействует с хранилищем строго через контракты (интерфейсы). Привязка интерфейсов к реализациям осуществляется в Service Provider.  
* **Слой доступа к данным (Repositories):** Изолирует работу с Eloquent ORM и SQL. Контроллеры и сервисы не знают о деталях реализации базы данных. Route-model binding не используется в пользу явных запросов через репозиторий.  
* **Паттерн Strategy:** Сортировка списка реализована через набор взаимозаменяемых стратегий с общим контрактом для избежания множественных ветвлений в сервисе.

### **2.3. Модель данных**

Основная таблица базы данных tasks.

| Поле | Тип данных | Описание / Ограничения   |
| :---- | :---- | :---- |
| id | BIGINT UNSIGNED | Первичный ключ. |
| title | VARCHAR(255) | Название задачи (обязательно). |
| title\_lower | VARCHAR(255) | Нормализованный заголовок для поиска; служебное поле, в API не отдаётся. |
| description | TEXT | Описание (опционально, nullable). |
| due\_date | DATETIME | Срок выполнения (в часовом поясе MSK). |
| create\_date | DATETIME | Доменная дата создания (передается клиентом). |
| status | VARCHAR(32) | Enum: 'не выполнена', 'выполнена'. |
| priority | VARCHAR(32) | Enum: 'низкий', 'средний', 'высокий'. |
| category | VARCHAR(255) | Строка с нормализацией (trim, приведение регистра). |
| created\_at, updated\_at | TIMESTAMP | Служебные метки Laravel. |

**Индексы (B-tree):** Одноколоночные индексы на поля `title_lower`, `due_date`, `create_date`, `status`, `priority`, `category`.

### **2.4. Спецификация API**

Все маршруты находятся в группе /api и принудительно возвращают ответы в формате JSON (настроен middleware force-json).

| Метод | Маршрут | Назначение | Успешный ответ   |
| :---- | :---- | :---- | :---- |
| **GET** | /tasks | Получение списка с пагинацией. Поддерживает query-параметры search, sort, page, per\_page. Метаданные передаются в заголовках. | 200 OK (Массив объектов) |
| **POST** | /tasks | Создание новой задачи. | 201 Created (id, message) |
| **GET** | /tasks/{id} | Получение информации о конкретной задаче. | 200 OK (Объект задачи) |
| **PUT** | /tasks/{id} | Полное обновление задачи. | 200 OK (message) |
| **DELETE** | /tasks/{id} | Удаление задачи. | 200 OK (message) |

### **2.5. Стандарты кодирования и контроля качества**

* **Статический анализ:** PHPStan \+ Larastan (Уровень 9). Код должен проходить проверку без ошибок и подавлений (без использования mixed в бизнес-логике).  
* **Стиль кода:** Строгое соблюдение PSR-1 и PSR-12 через инструмент Laravel Pint (пресет laravel).  
* **Документирование кода:** Использование семантически верного PHPDoc. Обязательны осмысленные описания параметров (@param), исключений (@throws) и уточнение дженериков для коллекций/массивов. Пустые теги без описания запрещены.  
* **Тестирование:** Полное покрытие функциональными и модульными тестами (Pest). Все методы API проверяются на положительные и отрицательные сценарии (ошибки валидации, отсутствие записей, граничные значения).  
* **CI/CD:** Проект снабжен GitHub Actions конвейером для автоматической проверки стиля, статического анализа и прогона тестов.

### **2.6. Принятые технические решения и ограничения**

* **Физическое удаление:** Реализован Hard Delete, так как интерфейс восстановления не предусмотрен в ВТЗ. В будущем возможен переход на SoftDeletes для ведения аудита.  
* **Нормализация категорий:** Категории хранятся как строки с нормализацией регистра и пробелов. Это упрощает текущую архитектуру, но при росте системы рекомендуется вынесение категорий в отдельную справочную таблицу.  
* **Поиск по названию:** Регистронезависимый поиск по полю `title_lower` (`LIKE '%search%'`). Поле заполняется при сохранении задачи; для индексации используется B-tree на `title_lower`. При ведущем `%` индекс на больших таблицах может не применяться.  
* **Кэширование:** Повторяющиеся чтения списка, поисковых выборок и отдельной задачи кэшируются в Redis. При изменении записи сбрасываются списки и поиск; кэш просмотра — только у затронутой задачи.  
* **Изоляция дат:** Доменная дата создания (create\_date) умышленно отделена от технических меток ORM (created\_at / updated\_at). Часовой пояс стандартизирован в MSK (Europe/Moscow).
