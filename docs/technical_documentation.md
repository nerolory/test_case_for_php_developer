# Технические решения проекта

## Введение

Документ описывает архитектурные решения, принятые в проекте, а также возможные направления его развития.

Цель документа — объяснить, почему система реализована именно так, какие преимущества это даёт, какие ограничения существуют и как проект может развиваться в дальнейшем.

Документ не является планом работ или списком обязательных изменений. Все рекомендации рассматриваются как возможные улучшения, необходимость которых зависит от требований к проекту, нагрузке и инфраструктуре.

## 1. Удаление записей

### Текущее решение

В настоящий момент используется физическое удаление записей из базы данных.

Такой подход выбран потому, что в системе отсутствует интерфейс для просмотра и восстановления удалённых задач. Хранить записи, которыми невозможно управлять, нецелесообразно.

### Преимущества

* Простая реализация.
* Отсутствие накопления неиспользуемых данных.
* Нет необходимости обслуживать архив удалённых записей.
* Минимальные требования к объёму базы данных.

### Возможные ограничения

При ошибочном удалении восстановить задачу невозможно.

Кроме того, при развитии проекта может появиться необходимость:

* хранить историю изменений;
* восстанавливать удалённые задачи;
* вести аудит действий пользователей;
* анализировать статистику удалений.

### Возможные улучшения

В таких случаях рекомендуется использовать мягкое удаление (`SoftDeletes`).

Преимущества такого подхода:

* возможность восстановления задачи;
* сохранение истории;
* отсутствие нарушения ссылочной целостности;
* сохранение непрерывности идентификаторов и временных меток.

При использовании мягкого удаления желательно определить срок хранения удалённых записей и настроить их автоматическую очистку.

## 2. Формат ответов API

### Текущее решение

Формат ответов зависит от типа запроса.

Получение одной задачи возвращает объект:

```json
{
  "id": 1,
  "title": "Task",
  "status": "done"
}
```

Получение списка возвращает массив объектов:

```json
[
  {},
  {},
  {}
]
```

Операции создания, изменения и удаления возвращают сообщение:

```json
{
  "message": "Task deleted successfully"
}
```

Такой подход прост и понятен, однако клиенту приходится поддерживать несколько различных форматов ответов.

### Возможные улучшения

Для публичного API удобнее использовать единый формат:

```json
{
  "data": {},
  "meta": {},
  "links": {}
}
```

Преимущества:

* одинаковая структура для всех эндпоинтов;
* единые правила сериализации;
* упрощение клиентской части;
* возможность безболезненно расширять ответы дополнительными данными.

Текущая реализация полностью соответствует примерам, приведённым в техническом задании.

Использование единого формата ответов рассматривается как возможное направление развития API при увеличении количества эндпоинтов и усложнении клиентской логики.

## 3. Пагинация списка

### Текущее решение

Список задач возвращается массивом, а информация о пагинации передаётся в HTTP-заголовках:

* X-Total-Count;
* X-Per-Page;
* X-Current-Page;
* X-Last-Page.

Такой подход работает корректно, но клиенту приходится отдельно обрабатывать тело ответа и его заголовки.

### Возможные улучшения

Более распространённым решением является возврат пагинации в теле ответа:

```json
{
  "data": [],
  "meta": {
    "current_page": 1,
    "last_page": 10,
    "per_page": 20,
    "total": 200
  }
}
```

Преимущества:

* вся информация находится в одном месте;
* формат проще документировать;
* уменьшается вероятность ошибок на стороне клиента.

## 4. Категории задач

### Текущее решение

Категория хранится строковым полем непосредственно в таблице задач.

Перед сохранением значение нормализуется:

* удаляются лишние пробелы;
* приводится к единому регистру;
* предотвращается создание очевидных дубликатов.

Такой подход прост и хорошо подходит для небольших проектов.

### Возможные ограничения

При развитии системы могут возникнуть сложности:

* одинаковые категории будут храниться многократно;
* невозможно хранить дополнительные свойства категории;
* сложно обеспечить целостность данных;
* строковые индексы занимают больше места.

Для текущего тестового задания использование строкового поля полностью соответствует требованиям и не создаёт ограничений.

### Возможные улучшения

Категории можно вынести в отдельную таблицу:

```text
categories
-----------
id
name
slug
color
icon

tasks
-----------
id
category_id
...
```

Преимущества:

* одна категория хранится в единственном экземпляре;
* появляется возможность добавлять дополнительные свойства;
* улучшается целостность данных;
* индекс по числовому идентификатору компактнее и работает быстрее.

## 5. Статус и приоритет

### Текущее решение

Статус и приоритет хранятся как строковые перечисления.

Например:

```text
Статус:
- не выполнена
- выполнена

Приоритет:
- низкий
- средний
- высокий
```

Такой подход удобен на этапе разработки и делает данные понятными при просмотре таблиц.

### Возможные ограничения

Строковые значения создают ряд проблем:

* переименование требует изменения данных;
* сложно реализовать многоязычность;
* индексы занимают больше места;
* появляется риск рассинхронизации значений.

Такой формат выбран в соответствии с примерами, приведёнными в техническом задании.

### Возможные улучшения

Предпочтительнее хранить машинные коды:

```text
Status:
pending
done

Priority:
low
medium
high
```

Человекочитаемые названия формируются через механизм локализации.

Такой подход:

* упрощает поддержку нескольких языков;
* уменьшает размер индексов;
* делает API более стабильным;
* позволяет менять отображаемые названия без миграции данных.

## 6. Даты и часовые пояса

### Текущее решение

Поле `create_date` является доменной датой и хранится отдельно от служебных полей Laravel:

* created_at;
* updated_at.

Поле принимается от клиента и сохраняется без изменений.

Дата создания задачи и срок выполнения (`due_date`) не сравниваются между собой. Система допускает ситуацию, когда задача считается выполненной в прошлом, а запись о ней создана позже.

### Причины выбора

Такой подход позволяет разделить:

* технические даты, которые относятся к записи в базе;
* бизнес-даты, которые относятся непосредственно к задаче.

Это делает модель данных более гибкой.

### Возможные улучшения

Для распределённых систем рекомендуется:

* хранить даты в UTC;
* передавать временную зону клиента явно;
* преобразовывать даты в локальное время только при отображении.

Это позволяет избежать ошибок, связанных с переходом между часовыми поясами и летним временем.

## 7. Привязка модели к маршруту (Route Model Binding)

### Текущее решение

В проекте не используется автоматическая привязка моделей Laravel (`Route Model Binding`).

В контроллер передаются только проверенные примитивные значения, например:

```php id="5rduyn"
public function show(int $id)
```

Получение и изменение данных выполняется через репозиторий.

### Причины выбора

Основная причина — наличие единой точки доступа к данным.

Если использовать Route Model Binding, Laravel выполнит запрос к базе ещё до вызова контроллера:

```php id="xfuqrc"
public function show(Task $task)
```

При использовании репозитория это часто приводит к повторному запросу или усложняет архитектуру.

Использование только идентификатора позволяет:

* сосредоточить работу с данными в одном месте;
* проще менять источник данных;
* легче внедрять кэширование;
* уменьшить связность между слоями приложения.

### Возможные ограничения

Такой подход требует более подробного описания маршрутов и немного большего объёма кода.

Кроме того, часть встроенных возможностей Laravel становится менее удобной.

При этом для проектов, не использующих отдельный слой репозиториев, Route Model Binding остаётся удобным и рекомендуемым инструментом Laravel.

## 8. Безопасность и контроль доступа

### Текущее решение

В настоящий момент API является открытым.

Маршруты не защищены middleware авторизации, а методы `authorize()` в Form Request возвращают:

```php id="pruvtw"
return true;
```

Пользовательская модель и принадлежность задач пользователям отсутствуют.

### Причины выбора

Такой подход упрощает:

* локальную разработку;
* тестирование через Swagger;
* проверку запросов через Postman;
* интеграцию на ранних этапах разработки.

### Возможные ограничения

Открытый API подходит только для внутренних проектов или демонстрационных решений.

При размещении в открытой сети появляются риски:

* просмотр чужих данных;
* изменение записей без ограничений;
* удаление данных;
* автоматизированный перебор идентификаторов.

### Возможные улучшения

Развитие системы может происходить поэтапно.

#### Этап 1. API Key

Простейшая защита реализуется через middleware, который проверяет:

```text id="hhl31j"
X-Api-Key
```

или

```text id="p9nfbq"
Authorization: Bearer <token>
```

Секрет хранится в `.env`.

Такой подход прост и подходит для сервисных интеграций.

#### Этап 2. Laravel Sanctum

Если в системе появляются пользователи, рекомендуется использовать Laravel Sanctum.

Он позволяет:

* выдавать персональные токены;
* отзывать токены;
* ограничивать права доступа;
* управлять сессиями и устройствами.

Маршруты защищаются middleware:

```text id="n6izhi"
auth:sanctum
```

#### Этап 3. Policies

После появления пользователей рекомендуется внедрить Policies.

Например:

```php id="3sv4iv"
return $task->user_id === $user->id;
```

Это позволяет контролировать:

* просмотр задач;
* изменение;
* удаление;
* выполнение административных действий.

Такой подход соответствует стандартной архитектуре Laravel и хорошо масштабируется.

## 9. Ограничение частоты запросов

### Текущее решение

Ограничение частоты запросов не используется.

Все запросы обрабатываются без лимитов.

### Возможные ограничения

При публикации API в интернете это может привести к следующим проблемам:

* DDoS-атаки;
* перебор идентификаторов;
* чрезмерная нагрузка на базу данных;
* исчерпание пула соединений.

### Возможные улучшения

Laravel предоставляет встроенный механизм ограничения частоты запросов.

Для API можно использовать middleware:

```text id="mmbwym"
throttle:api
```

Лимиты задаются централизованно через:

```text id="hjm9fr"
RateLimiter::for(...)
```

### Пример стратегии

Можно использовать разные лимиты:

* 60 запросов в минуту для анонимных клиентов;
* 300 запросов в минуту для клиентов с API Key;
* индивидуальные лимиты для пользователей Sanctum.

Ограничение частоты запросов рекомендуется использовать совместно с механизмами авторизации.

## 10. Фильтрация списка задач

### Текущее решение

В настоящий момент доступны:

* поиск по названию;
* сортировка списка.

Такого функционала достаточно для небольшого количества задач.

### Возможные ограничения

При увеличении объёма данных пользователям становится неудобно работать со списком.

Например:

* невозможно вывести только выполненные задачи;
* нельзя получить задачи определённой категории;
* отсутствует фильтрация по приоритету.

### Возможные улучшения

Рекомендуется поддержать фильтрацию по:

* статусу;
* приоритету;
* категории;
* диапазону дат;
* сроку выполнения.

Например:

```text id="ztoqsz"
/api/tasks?status=done

/api/tasks?priority=high

/api/tasks?category=work

/api/tasks?due_date_from=2026-01-01&due_date_to=2026-01-31
```

Это позволит значительно сократить объём данных, передаваемых клиенту.

## 11. Индексы и производительность базы данных

### Текущее решение

В таблице задач используются одноколоночные B-Tree индексы.

Индексируются поля:

* title_lower;
* due_date;
* create_date;
* status;
* priority;
* category.

Такой набор индексов хорошо подходит для текущего объёма данных и поддерживает основные сценарии работы.Поле `title_lower` добавлено для более удобной индексации при регистронезависимом поиске по названию: значение заполняется при сохранении задачи и не выводится в API.

### Возможные ограничения

При увеличении количества записей производительность может снижаться.

Особенно это заметно:

* при сложной фильтрации;
* при сортировке больших выборок;
* при поиске по нескольким полям одновременно.

### Возможные улучшения

Для часто используемых комбинаций полей рекомендуется создавать составные индексы.

Например:

```text id="ibjlwm"
(status, due_date)

(category, create_date)

(priority, due_date)
```

Это позволяет:

* уменьшить количество чтений;
* ускорить сортировку;
* снизить нагрузку на диск.

Перед созданием новых индексов желательно анализировать реальные запросы через:

```text id="6fcw8j"
EXPLAIN
```

Это позволяет избежать появления лишних индексов, которые замедляют операции записи.

Примечание: поиск выполняется через `title_lower LIKE '%search%'`, поэтому при ведущем `%` индекс на `title_lower` на больших выборках может не использоваться. Индексы на `due_date` и `create_date` соответствуют сортировке списка.

## 12. Поиск и масштабирование

### Текущее решение

Поиск выполняется через:

```sql id="6wzt3d"
title_lower LIKE '%search%'
```

Для этого в таблице задач добавлено поле `title_lower` с B-Tree индексом. Значение синхронизируется с `title` при сохранении записи.

Такой запрос обеспечивает регистронезависимый поиск без функции `LOWER()` в SQL.

### Возможные ограничения

При шаблоне `'%…%'` B-Tree индекс на `title_lower` не используется оптимизатором.

При росте таблицы база данных будет вынуждена просматривать все записи.

Это может привести к:

* увеличению времени ответа;
* росту нагрузки на процессор;
* появлению полного сканирования таблицы (`Full Table Scan`).

### Возможные улучшения

При увеличении объёма данных возможны несколько вариантов.

#### Full Text Search

Использовать встроенный полнотекстовый поиск СУБД.

Преимущества:

* высокая скорость;
* поддержка ранжирования результатов;
* поиск по нескольким словам.

#### Специализированные поисковые системы

Для сложных сценариев можно использовать:

* Meilisearch;
* Elasticsearch.

Они позволяют:

* искать по нескольким полям;
* использовать нечёткий поиск;
* учитывать опечатки;
* ранжировать результаты;
* строить поисковые подсказки.

Такой подход оправдан только при достаточно большом объёме данных и сложной логике поиска.

## 13. Кэширование

### Текущее решение

Повторяющиеся чтения задач кэшируются в Redis через стандартный механизм Laravel.

Кэшируются:

* список задач и поисковые выборки (`GET /api/tasks`, в том числе с параметром `search`);
* отдельная задача (`GET /api/tasks/{id}`).

При создании, изменении или удалении задачи сбрасываются все закэшированные списки и поисковые результаты. Кэш просмотра сбрасывается только у изменённой или удалённой записи; данные остальных задач в кэше сохраняются.

Если Redis недоступен или кэш отключён в настройках приложения, запросы выполняются напрямую к базе данных.

### Возможные ограничения

При росте числа одновременных пользователей общий сброс списков и поиска при любом изменении может увеличивать нагрузку на базу.

Разделение кэша по пользователям пока не предусмотрено.

### Возможные улучшения

* тегирование по пользователю после появления аутентификации;
* настройка времени жизни кэша под реальную нагрузку;
* выборочная инвалидация поисковых выборок без полного сброса списка.

## 14. Асинхронные операции и очереди

### Текущее решение

Все операции выполняются синхронно в рамках одного HTTP-запроса.

Это означает, что пользователь ожидает завершения всех действий, прежде чем получить ответ от сервера.

### Возможные ограничения

На начальном этапе это не вызывает проблем.

Однако со временем может потребоваться:

* отправка email-уведомлений;
* push-уведомления;
* синхронизация со сторонними сервисами;
* запись аудита;
* генерация отчётов;
* отправка сообщений в другие системы.

Если выполнять эти операции в HTTP-запросе, время ответа API начнёт расти.

### Возможные улучшения

Побочные действия рекомендуется выполнять асинхронно через очереди Laravel.

Основной запрос должен:

1. сохранить задачу;
2. опубликовать событие;
3. вернуть ответ клиенту.

Дополнительные действия выполняются отдельными воркерами.

### Возможные драйверы очередей

Для хранения очередей могут использоваться:

* Redis — наиболее распространённое решение;
* RabbitMQ — подходит для сложной маршрутизации сообщений;
* Amazon SQS — для облачной инфраструктуры.

Использование очередей позволяет:

* сократить время ответа API;
* повысить устойчивость системы;
* независимо масштабировать обработчики фоновых задач.

## 15. Идемпотентность операций создания

### Текущее решение

Каждый POST-запрос на создание задачи обрабатывается независимо.

Если клиент повторно отправит одинаковый запрос, система создаст несколько записей.

Причиной могут быть:

* сетевые ошибки;
* повторная попытка клиента;
* двойное нажатие кнопки;
* повторная доставка сообщения прокси-сервером.

### Возможные ограничения

Появление дубликатов может привести к:

* искажению данных;
* повторной отправке уведомлений;
* ошибкам в интеграциях;
* усложнению бизнес-логики.

### Возможные улучшения

Для защиты от повторной обработки запроса можно использовать заголовок:

```text id="qbtcfv"
X-Idempotency-Key
```

Алгоритм работы:

1. клиент генерирует уникальный ключ;
2. сервер проверяет наличие ключа в Redis;
3. если ключ уже существует — возвращается сохранённый ответ;
4. если ключ отсутствует — запрос выполняется;
5. результат сохраняется с ограниченным временем жизни.

Например:

```text id="m3myvh"
TTL = 24 часа
```

Такой механизм гарантирует, что один и тот же запрос не будет выполнен повторно.

## 16. Версионирование API

### Текущее решение

Маршруты зарегистрированы без указания версии:

```text id="mjlwmf"
/api/tasks
```

Это упрощает разработку и уменьшает количество маршрутов.

### Возможные ограничения

При изменении структуры API может быть нарушена обратная совместимость с существующими клиентами.

Например:

* изменится формат ответа;
* появятся обязательные поля;
* изменятся правила валидации;
* изменится логика пагинации.

В результате существующие клиенты могут перестать работать или потребуют доработки.

### Возможные улучшения

Для публичных API рекомендуется использовать версионирование:

```text id="drn3lh"
/api/v1/tasks

/api/v2/tasks
```

Такой подход позволяет:

* поддерживать несколько поколений API;
* безопасно изменять контракт;
* постепенно переводить клиентов на новую версию;
* уменьшить количество критических изменений.

## 17. Стандартный каркас Laravel

### Текущее решение

В проекте присутствуют стандартные артефакты Laravel:

* модель `User`;
* миграция `users`;
* `UserFactory`;
* настройки аутентификации;
* базовая инфраструктура авторизации.

В текущей реализации они не используются.

### Причины выбора

Стандартный каркас оставлен намеренно.

Это позволяет:

* быстрее внедрить авторизацию;
* подключить Laravel Sanctum;
* добавить личные кабинеты;
* реализовать разграничение доступа;
* использовать готовую инфраструктуру Laravel.

### Возможные ограничения

Если проект останется полностью анонимным API, часть этого кода будет неиспользуемой.

Это может:

* усложнять навигацию по проекту;
* создавать ощущение незавершённой функциональности;
* вводить в заблуждение новых разработчиков.

### Возможные улучшения

В дальнейшем возможны два подхода.

#### Сохранить каркас

Подходит, если планируется:

* регистрация пользователей;
* авторизация;
* персональные токены;
* разграничение прав доступа.

#### Удалить каркас

Подходит, если проект останется небольшим сервисом без пользователей.

В этом случае можно:

* удалить модель `User`;
* удалить миграцию `users`;
* упростить конфигурацию аутентификации;
* убрать неиспользуемые зависимости.
